# AI 知识与实践平台联合升级设计

## 1. 背景与决策

项目已经从少量工具教程发展为同时覆盖模型使用、研究、智能体、视觉创作、开发与自动化的 AI 知识站。现有首页、文档阅读系统和旗舰内容已经形成基础，但项目治理、品牌配置、动效性能、阅读控件和发布验证仍分散在不同文件中，继续局部叠加功能会放大维护成本。

本轮采用“治理统一 + 性能稳定 + 阅读增强”的联合方案。它不继续堆叠展示库，也不把站点收缩成传统文档模板，而是先建立一个可长期扩展的产品底座，再让首页和文章中的视觉与交互服务于真实阅读任务。

品牌名称暂时保留“AI 使用教程”，避免破坏仓库、域名和既有认知；对外产品描述统一升级为“AI 知识与实践中心”。未来如需正式改名，应单独进行品牌迁移，不在本轮隐式完成。

## 2. 目标

1. 建立唯一站点配置源，统一产品、作者、仓库、导航和社交信息。
2. 把 `AGENTS.md`、`.codex/`、项目文档和执行脚本整理成可发现、可复用、可验证的协作系统。
3. 保留首页的品牌动效，同时消除持续渲染、重复引擎和 JavaScript 失败导致的空白或卡顿风险。
4. 为文档页补齐高价值阅读控件，统一导航、代码、图表、目录和文章翻页体验。
5. 建立内容类型、组件目录、研究溯源和发布质量门，避免文章再次变成同质化模板或无依据的视觉堆叠。
6. 保持 Next.js 静态导出和 GitHub Pages 部署兼容。

## 3. 非目标

- 不引入账户、云端收藏、评论、在线 AI 问答或私有后端。
- 不为了“展示能力”同时加载 Three.js、Matter.js、Spline、Rive、MapLibre 等所有库。
- 不在本轮重写六篇旗舰长文正文，也不迁移现有公开 URL。
- 不建立完整 Storybook、CMS 或编辑器后台；组件目录先使用仓库文档和站内示例。
- 不把字体大小、云端笔记、跨设备书签作为首批功能。
- 不自动提交当前工作区中与本规格无关的历史修改。

## 4. 现状证据与约束

### 4.1 配置分散

站点名称、作者和联系信息当前分布在 `README.md`、`CONTRIBUTING.md`、`package.json`、根布局、首页、`src/lib/shared.ts` 和 `src/lib/site-content.ts` 中。多处手工维护会导致页面标题、首页文案和作者卡片不一致。

### 4.2 首页运行时过重

首页同时存在 GSAP、ScrollTrigger、Lenis、Three.js 和持续帧循环。审查期间浏览器曾出现首页截图超时与标签页关闭，不能据此断言所有设备都会崩溃，但足以把运行时稳定性列为发布阻断项。`physics-knowledge-field.tsx` 与 `knowledge-map-chart.tsx` 当前没有进入有效产品路径；Matter.js 也只服务于未使用组件。

### 4.3 文档能力缺口

知识库已经具有三级导航、页内目录、提示词、资料折叠、Mermaid 和 ECharts，但仍缺少阅读进度、返回顶部、图表全屏、代码换行/展开、清晰的上一篇/下一篇关系以及更克制的移动端操作。

### 4.4 工作流偏文章模板化

现有 `.codex/workflows/new-tutorial.md` 对所有文章使用相近的步骤和验收结构，容易生成操作手册式内容。项目尚缺研究型文章、内容重构、阅读体验、性能动效和发布流程的明确路由，也没有统一的文章简报、研究计划、组件简报和发布报告模板。

## 5. 目标架构

### 5.1 单一配置源

新增 `src/config/site.ts`，导出只读的站点配置对象和必要类型，至少包含：

- 品牌名称、产品描述、默认 SEO 标题与描述；
- 站点 URL、仓库 URL、Pages 基础路径；
- 作者姓名、个人简介、微信号、公众号名和二维码资产；
- 顶栏、页脚和主要文档入口；
- 社交链接和版权信息。

根布局、首页、文档入口、作者卡片和共享布局只消费该配置，不再复制字符串。内容文章可以引用品牌概念，但不得把运行时配置反向散落到 MDX 中。

`src/lib/shared.ts` 与 `src/lib/site-content.ts` 在迁移完成后删除或降为从 `site.ts` 派生的兼容导出；不得长期维护两份真相。

### 5.2 文档职责

- `AGENTS.md`：项目级最高协作规则，负责范围、任务路由、禁止事项、研究来源、静态导出、UI/性能门和验证命令。
- `.codex/README.md`：面向执行者的任务路由表，只回答“当前任务应该读哪个工作流/模板”。
- `.codex/workflows/`：按任务类型描述输入、步骤、检查点和交付物。
- `.codex/templates/`：存放可复制的内部执行简报与报告模板。
- `docs/content-model.md`：定义教程、解释文、案例、参考、实验五种公开内容类型。
- `docs/component-catalog.md`：定义可复用 MDX/阅读组件、适用场景、反例和无障碍约束。
- `docs/qa/`：保存带日期、提交、范围与结果的 QA 报告；旧的 `design-qa.md` 归档到 `docs/qa/archive/`，不保留永久“已通过”断言。

### 5.3 工作流与模板

保留并改写现有工作流，同时补充：

- `research-and-write-article.md`：多来源研究、证据矩阵、事实核验、图片授权与成稿。
- `review-and-refactor-content.md`：发现模板化、失效来源、结构冗余和不可验证结论。
- `enhance-reading-experience.md`：文章控件、MDX 组件、响应式与无障碍增强。
- `performance-and-motion-audit.md`：动效预算、降级模式、内存/帧循环和浏览器验证。
- `release.md`：变更边界、自动验证、浏览器 QA、提交、推送和部署检查。

新增模板：

- `article-brief.md`：读者、问题、体裁、范围、期望证据和视觉需求。
- `research-plan.md`：研究问题、优先一手来源、时效日期和交叉验证。
- `component-brief.md`：用户价值、状态、响应式、无障碍、性能和静态导出约束。
- `release-report.md`：提交、变更范围、验证结果、已知限制和部署 URL。

内部作者提示词放在 `.codex/templates/`；提供给读者复用的提示词继续放在 `content/docs/reference/`。两者不得混用。

## 6. 内容模型与研究规范

### 6.1 五种内容类型

`docs/content-model.md` 定义以下体裁，并允许 frontmatter 增加 `contentType`：

1. `tutorial`：完成一个明确任务，步骤和结果可验证。
2. `explainer`：解释概念、判断框架和边界，不强制步骤结构。
3. `case-study`：以问题、约束、选择、结果和复盘展开。
4. `reference`：可检索的术语、模板、API 或对照表。
5. `experiment`：记录假设、方法、观察、限制和后续验证。

现有文章可逐步补充该字段；首批迁移不得因为缺少字段导致构建失败。新文章必须声明类型，工作流根据类型选择结构，不再统一生成“步骤 + 验收清单”。

### 6.2 来源与视觉

- 易变事实优先引用官方文档、研究论文或产品公告，并写明核验日期。
- 社交媒体或公众号文章只能作为选题线索、案例或观点来源，不能替代关键事实的一手来源。
- 同类热门文章可以用于比较叙事和读者问题，但不得复刻其标题、段落结构或独特表达。
- 下载的图片必须记录来源、授权或合理使用理由；无法确认时使用自制图、代码生成图或 ImageGen 素材。
- 生成图和示意数据必须明确标注“示意”或“生成”，不得伪装成真实结果。
- 图片必须有有效替代文本；纯装饰图使用空替代文本并从辅助技术中隐藏。

## 7. 首页性能与动效系统

### 7.1 动效模式

首页提供三个明确模式：

- `auto`：默认模式，遵循设备能力和系统 `prefers-reduced-motion`。
- `reduced`：保留淡入和状态反馈，关闭平滑滚动、视差和持续 WebGL。
- `off`：关闭非必要动效，展示完整静态页面。

用户选择使用版本化 `localStorage` 保存。首次访问不覆盖系统偏好；JavaScript 不可用时，所有核心内容、导航与 CTA 仍然可读可点。

### 7.2 运行预算

- 首屏最多保留一个 WebGL 上下文。
- `KineticKnowledgeScene` 仅在可见且模式允许时运行；页面隐藏或离开视口后暂停。
- 设备像素比设置上限，低功耗/触摸设备降低粒子数量和帧率目标。
- Lenis 在触摸设备、reduced/off 模式和初始化失败时不启用。
- GSAP、Three.js 和重型实验模块通过动态导入进入客户端路径。
- 避免同时存在多个常驻 `requestAnimationFrame`、GSAP ticker 和库级循环；由统一控制器负责启动与清理。
- 每个 effect 必须在卸载时移除监听、ticker、ScrollTrigger、WebGL 资源和定时器。

### 7.3 降级与删除

- WebGL、动态导入或运动控制失败时显示稳定的 CSS 背景和静态视觉，不显示空白画布。
- 首页不得以正文透明度或 hydration 完成作为内容可见前提。
- 删除没有真实入口且无复用计划的 `physics-knowledge-field.tsx`、`knowledge-map-chart.tsx` 及其专用依赖；若实现核查发现被其他页面消费，则保留并补充入口与测试。
- Matter.js 只有在存在明确的交互任务和性能预算时才保留，不以“已安装”为理由制造展示区。

### 7.4 验收阈值

- 桌面与 390px 移动视口无水平溢出。
- 首页 `auto`、`reduced`、`off` 三种模式均可独立刷新并稳定显示。
- 页面切到后台 5 秒后无持续 WebGL 绘制；回到前台可以恢复。
- 控制台无未处理异常、重复初始化警告和站内 RSC 预取 404。
- 关闭 JavaScript 后仍能阅读品牌主张、推荐内容、作者信息和页脚链接。

## 8. 文档阅读控件

### 8.1 第一批控件

1. `ReadingProgress`：顶部细进度条，仅反映正文阅读进度，不遮挡站点顶栏。
2. `BackToTop`：滚动超过一个正文视口后出现；桌面靠近正文右下，移动端避开底部安全区和目录按钮。
3. `DiagramViewer`：为 Mermaid 和数据图提供全屏/退出、缩放复位与键盘 Escape；移动端使用可滚动全屏层。
4. `CodeToolbar`：复制反馈、长行换行切换、超高代码折叠/展开；不改写 Shiki 输出语义。
5. `ArticlePager`：严格按照知识树提供上一篇/下一篇，显示所属分组和标题，不按磁盘目录随机跳转。

### 8.2 目录与移动端

- 桌面右侧目录折叠后使用无边框悬浮按钮，并提供固定标签或 tooltip，避免只有含义不明的图标。
- 移动端目录入口显示“本文导航”及当前阅读进度，不复制桌面侧栏控件。
- 移动端主要操作只保留目录、返回顶部和必要分享/打开动作；“更多打开方式”降为次级菜单。
- 左侧知识树继续保持“开始、专题、工程”的固定顺序，当前文章所属组自动展开，其他组保留用户折叠状态。

### 8.3 状态与无障碍

- 所有按钮使用真实 `button`、可见焦点、明确 `aria-label` 和必要的 `aria-expanded`。
- 全屏查看器启用焦点管理，关闭后把焦点返回触发按钮。
- 动画遵循 reduced/off 模式；阅读功能不得依赖动画完成。
- 隐藏的抽屉、目录或覆盖层不得保留可聚焦元素，也不得被屏幕阅读器重复读取。
- 本地状态只保存界面偏好，不保存文章正文、个人信息或分析追踪数据。

### 8.4 统一富媒体外壳

图片、视频、图表、数据表、公式和可下载附件统一使用 `MediaFrame` 语义外壳，而不是各自创建一套标题栏。外壳提供：

- 洞察型标题，而不是“图 1”“示意图”等无信息标题；
- 一句话说明、来源、核验日期、数据口径和“示意/生成”状态；
- 按内容能力显示的精简工具栏；
- 标准、宽幅和全宽三种版面宽度；
- 加载、空数据、失败、离线和静态降级状态；
- 打印时自动移除交互控件，保留标题、图像、关键值、来源与注释。

工具栏不得永久堆满所有按钮。桌面默认显示一至三个高频动作，其余进入“更多”；移动端使用底部动作面板。只有内容实际支持时才显示放大、下载、复制或导出。

### 8.5 图片与图片组

新增 `ArticleImage`、`ImageGallery` 和共享 `LightboxViewer`：

- 图片必须声明原始尺寸或稳定宽高比，避免加载时正文跳动。
- 使用响应式资源和正确 `sizes`，首屏主图可预加载，正文图片默认懒加载。
- 单击图片或“查看大图”进入灯箱；支持放大、缩小、100%、适应屏幕、复位、拖拽和平移。
- 桌面支持滚轮缩放和键盘 `+`、`-`、`0`、方向键、Escape；触摸端支持双击、捏合和拖动。
- 图片组支持上一张/下一张、缩略导航和当前位置，不自动轮播。
- 灯箱打开后锁定背景滚动、限制焦点；关闭后焦点返回原图。
- 可选择显示“打开原图”与“下载”，但受版权限制的远程图片不得默认提供下载。
- 截图型教程可增加标注点和局部放大，但热点必须有文字列表作为无障碍替代。
- “前后对比”使用专门的 `ImageCompare`，提供键盘可操作滑杆和两个独立查看入口，不把关键差异藏在拖动后。

二维码不进入通用灯箱：作者二维码继续使用明确的联系卡片和预览层，避免误触后占满屏幕。

### 8.6 视频与动态演示

新增 `ArticleVideo`，优先使用原生 HTML5 视频能力，并遵守：

- 不自动播放带声音的视频，不用背景视频替代正文说明。
- 必须提供海报图、标题、时长、字幕或文字稿；教学视频应提供章节时间点。
- 本地视频使用 WebVTT 字幕；第三方视频只有在用户点击后才加载 iframe，并显示平台与隐私提示。
- 提供播放/暂停、静音、进度、倍速、字幕、画中画和全屏；不重写浏览器已成熟的核心媒体控制。
- 进入全屏或影院模式时保留标题、章节和文字稿入口。
- 移动网络和 reduced-data 环境只加载海报，等待用户主动播放。
- 无法播放时显示文字稿、下载链接或外部原始页面，不留下空黑框。
- 短循环演示可使用静音视频，但必须有暂停按钮和静态替代图；不以 GIF 承载长时间、高分辨率教程。

### 8.7 图表、数据图与交互可视化

图表按分析任务选择最简单的可信图形：比较/排名优先条形图，时间变化优先折线或小多图，分布优先点图/直方图/箱线图，组成只在类别较少时使用环图，关系和流程优先 Mermaid 或结构化示意。3D、粒子和动画只有在表达规模、流动或空间关系时才允许使用。

新增 `ChartFrame`，并让现有 `SourceMixChart` 迁入统一外壳：

- 标题直接写出要点，图下注明来源、样本、时间范围、单位、缺失值和口径。
- 关键数值与标签无需 hover 即可看到；hover 只提供次级明细。
- 颜色角色固定为中性背景、主焦点、比较色、选择/警告状态，并提供形状、线型或直接标签作为冗余编码。
- 提供全屏、复位视图、查看数据和可选导出；探索型图表才提供缩放、筛选与图例开关。
- “查看数据”展示可访问表格；允许公开复用的数据可下载 CSV，示意数据必须在表格和导出文件中继续标注。
- 导出优先 SVG，Canvas 图表可导出 PNG；导出内容必须带标题、来源、日期与必要注释，不能只导出失去上下文的图形。
- URL 只保存对理解有价值的筛选、时间范围或选中项；短暂 hover 和纯视觉缩放不写入 URL。
- 移动端不缩小桌面图表硬塞入正文：改为更少刻度、直接标签、纵向排列或可控横向滚动，且触摸目标不小于 44px。
- ECharts 只由图表组件动态加载，每个实例在离开页面时销毁；首屏之外的实例接近视口后才初始化。
- 每个交互图表必须有静态或文本降级；打印与禁用 JavaScript 时至少保留洞察、关键值、数据表和来源。

### 8.8 公式与技术符号

新增 Markdown 数学处理和 `FormulaBlock`：

- 行内公式与块级公式使用 LaTeX 源码，经构建期插件输出 KaTeX/MathML；不得用公式截图。
- 块级公式可显示编号、变量解释和“复制 LaTeX”，长公式可横向滚动但不得裁切。
- 屏幕阅读器优先读取 MathML 或明确的文字解释；复杂公式正文后必须解释变量和结论。
- 灯箱式公式查看仅用于超宽推导，支持放大、缩小、复位和复制；普通公式不增加多余工具栏。
- 主题切换后公式颜色与正文一致，危险色、品牌色不得改变数学语义。

### 8.9 表格、引用与附件

- 普通表格使用滚动阴影、首列/表头语义、数字右对齐、单位放在列名中；不使用整表 `white-space: nowrap` 破坏长文本。
- 宽表格使用 `DataTable` 提供全屏、列显示、复制与 CSV 下载；排序和筛选只在数据量确实需要时启用。
- 移动端优先保留表格结构并横向滚动；只有字段明确且不会丢失比较关系时才转换为卡片。
- 引用分为作者原话、编辑摘要和警示说明三种视觉语义；原话必须带来源，编辑摘要不得使用引号伪装成引用。
- `DownloadCard` 显示文件名、格式、大小、用途、更新时间和来源，下载链接必须指向静态公开资产。
- PDF、表格或模板可以提供浏览器预览入口，但正文中同时给出文件信息和直接下载，预览失败不能阻断访问。

### 8.10 文章版式与内容控件

- 正文建立紧凑、标准、宽幅三条阅读通道：段落保持标准宽度，表格/图表可进入宽幅，真正需要空间的媒体才全宽。
- H2、H3、段落、列表、图注、控件之间使用统一垂直节奏；连续卡片不得把文章切割成仪表盘。
- 提供 `Callout`、`Definition`、`Quote`、`Steps`、`Tabs`、`Accordion`、`Footnote` 和 `Kbd` 的明确语义与样式，不允许仅为“好看”随意混用。
- 折叠控件只承载补充资料、源码和细节，不隐藏完成任务所需的核心步骤。
- 标签页在打印、无 JavaScript和复制正文时展现全部内容；移动端标签可横向滚动并显示边缘提示。
- 文章头部可显示类型、难度、更新时间、预计阅读时间和核心成果；不重复出现多组同义元信息。
- 文末统一提供资料与注释、上一篇/下一篇、反馈入口和更新时间，不再堆叠页面级验收清单。
- 增加专用打印样式：展开关键折叠内容、保留 URL/来源、分页避免切断图表标题与图体、隐藏浮动按钮和导航外壳。

## 10. 测试与发布质量门

### 10.1 脚本

在 `package.json` 增加聚合命令 `npm run check`，按稳定顺序运行：

```text
npm test
npm run lint
npm run types:check
npm run build
```

聚合脚本不替代单项命令，便于定位失败。

### 10.2 自动检查

- 站点配置测试：品牌、作者、仓库和导航必填，消费端不再复制核心字符串。
- 内容模型测试：新文章的 `contentType`、frontmatter、日期和允许值合法；旧内容在迁移期兼容。
- 导航测试：无重复 URL、死链接、空分组和孤立旗舰文章。
- 资产测试：本地图片存在、路径正确、非装饰图片有 alt；来源记录按内容工作流检查。
- 阅读组件测试：复制、换行、展开、全屏、Escape、上一篇/下一篇和状态持久化。
- 富媒体测试：灯箱缩放/平移/切换、焦点锁定与返回、视频降级、公式输出、表格滚动和附件链接。
- 可视化测试：关键值无需 hover 可见、查看数据表可用、示意/真实数据标签正确、图表销毁和静态降级有效。
- 动效测试：reduced/off 不初始化持续渲染；组件卸载会清理监听与循环。
- 静态产物检查：关键首页和文档 URL 在 `out/` 中存在。

### 10.3 浏览器 QA

发布前至少检查：

- 1440px 桌面与 390px 手机；
- 首页和一篇含代码、Mermaid、ECharts 的旗舰文章；
- 一篇包含图片组、视频海报、公式、宽表格和附件的富媒体测试文章；
- 浅色、深色、reduced motion 和 motion off；
- 键盘导航、焦点返回、侧栏/目录开合和全屏退出；
- 控制台错误、水平溢出、主题闪烁和站内链接；
- GitHub Pages 基础路径下的静态资源和路由。

QA 结果写入基于 `release-report.md` 的一次性报告，必须包含日期、提交哈希、浏览器/视口、通过项、失败项和已知限制。

## 11. 实施阶段

### 阶段一：治理与单一配置源

更新 `AGENTS.md` 与 `.codex/` 路由，新增模板、内容模型和组件目录；建立 `src/config/site.ts` 并迁移消费端；归档陈旧 QA 文档。该阶段不改变公开 URL。

### 阶段二：首页稳定性

重构首页动效生命周期和三种模式；动态加载重型库；删除确认无用的组件与依赖；建立静态降级和动效测试。

### 阶段三：阅读控件

按 ReadingProgress、BackToTop、DiagramViewer、CodeToolbar、ArticlePager 的顺序实现。每个控件完成单元测试和桌面/移动浏览器检查后再进入下一个，避免一次叠加多个未知交互。

### 阶段四：富媒体与文章样式

先实现共享 `MediaFrame`，再依次接入图片灯箱、视频、图表、公式、数据表和附件。使用一篇内部富媒体测试文章覆盖全部状态，确认桌面、移动、键盘、触摸、打印和无 JavaScript 降级后，再把能力用于公开文章。组件完成不代表每篇文章必须使用；作者根据内容价值选择。

### 阶段五：质量门与发布

增加 `npm run check`、内容/资产/静态产物检查，执行完整浏览器 QA，生成发布报告。只提交本轮明确范围内的文件；当前工作区既有变更需要单独核对归属后再决定是否一并发布。

## 12. 迁移与兼容策略

- 先增加新配置和兼容导出，再逐个迁移消费端，最后删除旧常量，避免半迁移状态。
- `contentType` 使用渐进迁移，新建强制、既有兼容；全部补齐后再升级为全库强制。
- 阅读控件以增强层接入，失败时保留原生代码、图表静态容器、目录和文章链接。
- 删除依赖前使用静态引用搜索、测试与生产构建共同确认无消费方。
- 不手工编辑 `.next/`、`out/` 或 `.source/`。

## 13. 风险与控制

### 13.1 脏工作区误提交

当前工作区包含上一轮内容与阅读系统变更。每个提交必须使用显式路径暂存，并在提交前运行 `git diff --cached --name-status`；禁止使用无范围的 `git add .`。

### 13.2 动效重构造成视觉倒退

保留现有静态品牌层作为基线，先建立降级模式与截图，再调整动画。性能优化不得通过删除所有品牌视觉完成。

### 13.3 控件挤占阅读空间

新控件优先合并进现有目录、代码和图表容器。桌面固定控件最多保留两个，移动端固定控件最多保留目录与返回顶部，避免形成工具条墙。

### 13.4 自动检查过度脆弱

内容规则只检查可客观判断的字段、链接和结构，不用正则强迫文章使用统一章节。性能预算先验证循环、加载和错误等稳定指标，不把易波动的单次 Lighthouse 分数作为唯一发布门。

## 14. 完成标准

本轮只有同时满足以下条件才视为完成：

1. 品牌、作者、仓库和导航只有一个配置真相，README、布局、首页与文档入口一致。
2. `AGENTS.md`、工作流、模板、内容模型和组件目录能够覆盖研究、写作、重构、视觉增强、性能审查与发布。
3. 首页三种动效模式稳定，无水平溢出、未处理错误、后台持续渲染或 JavaScript 失败空白。
4. 五个第一批阅读控件在桌面和移动端可用，键盘和 reduced motion 行为正确。
5. 图片、视频、图表、公式、表格和附件使用统一富媒体外壳；灯箱、全屏、缩放、降级、来源和移动交互符合本规格。
6. 图表具有洞察标题、数据口径、直接标签、可访问数据表和适当导出，不依赖 hover 才能理解。
7. 文章版式、语义控件和打印样式统一，富媒体不会把正文切割成卡片墙。
8. 未使用的首页组件和专用依赖完成核查并删除，或有明确保留理由、入口与测试。
9. `npm run check` 通过，静态产物关键路由存在。
10. 完成浅色、深色、桌面、移动、reduced/off 的浏览器 QA，并生成带提交哈希的发布报告。
11. Git diff 经过范围审查，未误提交用户现有但不属于本轮的修改。
