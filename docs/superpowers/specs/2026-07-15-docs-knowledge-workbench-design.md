# AI 文档知识工作台设计规格

## 目标

把现有 Fumadocs 文档区升级为与官网同一品牌的中文 AI 知识工作台。设计以视觉方案 2 的高密度、任务导向结构为骨架，融合视觉方案 3 的编辑感、知识地图和轻量品牌装饰，同时保留长文阅读效率、静态导出能力与未来内容扩展空间。

成功标准：

- `/docs/` 是独立的知识门户，而不是一篇放大的普通文章。
- 文档首页、分类页和文章页共享统一的暖米白、深墨色、紫色品牌体系，并有完整深色主题。
- 桌面端具备清晰的顶栏、分组侧栏、主内容区和右侧上下文栏；移动端只有一个顶栏且无横向溢出。
- 搜索、导航、主题切换、目录和主要入口真实可用，不使用静态假按钮。
- 最近更新、文章数量、标签和难度来自真实 MDX 数据，不伪造内容规模。
- 文档阅读页不加载首页的 Three.js、Matter.js、GSAP 或 Lenis 动效引擎。

## 用户与核心任务

主要用户是希望学习、比较或实际应用 AI 工具的中文读者，包括创作者、知识工作者和开发者。进入文档区后，他们需要完成四类任务：

1. 从零完成第一个 AI 任务。
2. 按工具找到可靠用法。
3. 按工作场景复用流程。
4. 沿系列课程建立体系化能力。

门户首页以这四类任务组织入口，不以技术分类或虚构统计为优先。

## 视觉方向

### 结构基准

采用已生成的第 2 个视觉方案作为结构基准：

- 64–72 像素的紧凑全局顶栏。
- 约 240 像素的分组左侧导航。
- 中央内容区优先呈现搜索、任务入口、推荐路径和最近更新。
- 约 220 像素的右侧上下文栏，用于本页目录、学习路径和真实更新。
- 小圆角、细边框、较高信息密度，不使用巨型欢迎卡。

### 品牌基准

采用已生成的第 3 个视觉方案作为品牌基准：

- 暖米白背景、深墨色正文、紫色主强调。
- 青柠只用于状态点、进度和极小标签；珊瑚色只用于少量提示。
- 编辑型排版、编号、细线和克制的几何知识地图装饰。
- 标题与正文具有明显层级，但不把所有内容放进圆角卡片。

### 禁止模式

- 不复制营销首页的大字号、滚动劫持、物理引擎或 3D 场景。
- 不使用玻璃拟态、霓虹赛博朋克、无限渐变或卡片套卡片。
- 不用“128 篇文章”“40% 进度”等无法从真实数据获得的数字。
- 不用纯装饰的控制按钮；可点击控件必须有真实行为。

## 信息架构

### 顶栏

顶栏包含品牌、快速开始、工具教程、系列课程、场景打法、参考资料、搜索、主题切换和 GitHub 入口。在文档区不显示重复的“进入文档”按钮。

### 左侧导航

左侧导航由 Fumadocs 页面树生成，分为：

- 从这里开始
- 工具教程
- 系列课程
- 场景打法
- 单篇笔记
- 参考资料

每组可折叠，当前项具有明确的紫色选中状态。底部保留“返回官网”和 GitHub 入口。桌面可收起，移动端使用模态抽屉并正确管理焦点和背景可访问性。

### 文档门户 `/docs/`

门户按以下顺序呈现：

1. 紧凑欢迎区：标题、一句话价值说明、全局搜索。
2. 四个任务入口：快速开始、按工具学习、解决场景问题、浏览系列课程。
3. 推荐学习路径：以五步语义化路径展示入门准备、工具选择、核心技能、进阶应用、实战落地。
4. 精选内容：从真实文章中选取高价值内容，不显示空占位卡。
5. 最近更新：按 `published` 排序，展示日期、分类、难度和标签。
6. 右侧上下文：本页导航、当前知识地图阶段和最近更新摘要。

知识地图可以使用语义化步骤组件；若加入关系图，只在门户动态导入现有 ECharts。装饰性图形必须是实际资源或图标库图标，不手写复杂 SVG。

### 分类首页

分类页保留 MDX 可编辑能力，但使用统一的分类头部和文章索引组件。索引优先显示真实文章；尚未完成的主题可放入“规划中”列表，必须明确标注为规划内容且不可伪装成可阅读文章。

### 文章阅读页

文章页标题区比当前高度减少约 40%，包含：

- 分类路径
- 标题与说明
- 发布时间、难度、标签
- 复制 Markdown、查看源码等紧凑操作

正文维持 42–48rem 的舒适行宽。右侧显示中文“本页目录”和轻量阅读进度。标题、代码块、表格、引用、提示框和步骤列表必须在浅色与深色主题下保持足够对比度。

## 内容数据模型

扩展 `source.config.ts` 的页面 schema，正式支持：

- `tags: string[]`
- `published: YYYY-MM-DD` 字符串
- `difficulty: 入门 | 进阶 | 专家`
- `featured: boolean`，可选
- `status: published | planned`，默认 `published`

构建时从 `source.getPages()` 生成文章摘要、真实数量、最近更新和精选内容。`published` 必须通过 schema 校验为 `YYYY-MM-DD`；格式错误直接令类型生成或构建失败，避免悄悄发布错误元数据。

## 首批新增内容

新增 8 篇可实际阅读的教程，每篇满足项目的 frontmatter 和写作标准：

1. `start-here/first-ai-task.mdx`：用“提问—校验—改写”完成第一个 AI 任务。
2. `tools/chatgpt/workspace-guide.mdx`：ChatGPT 项目式工作区入门，避免描述易变价格。
3. `tools/codex/repository-workflow.mdx`：使用 Codex 完成一次可验证的仓库任务。
4. `playbooks/research-to-brief.mdx`：从资料研究到结构化简报的工作流。
5. `playbooks/build-personal-knowledge-base.mdx`：搭建个人 AI 知识库的静态可复用流程。
6. `notes/prompt-debugging-checklist.mdx`：提示词失效时的诊断清单。
7. `reference/prompt-brief-template.mdx`：可复用的任务简报模板。
8. `series/ai-workflow-foundations/index.mdx`：AI 工作流基础系列入口，链接上述相关内容。

不新增与当前内容无关的大量占位文件。未来主题只在对应分类页的“规划中”列表中显示标题和目标。

## 组件边界

- `DocsPortal`：组合门户各模块，只接收已规范化的数据。
- `DocsPortalSearch`：打开现有 Fumadocs 搜索对话框。
- `LearningDestinations`：四个任务入口。
- `KnowledgePath`：五步学习路径，可访问、无重型动画。
- `ContentFeed`：精选与最近更新列表。
- `DocsArticleHeader`：文章元信息和操作栏。
- `DocsMeta` 工具：解析、排序和筛选页面数据，不依赖 React。

组件放在 `src/components/docs/`，数据工具放在 `src/lib/docs-content.ts`。文档专属样式放在 `src/app/docs/docs-theme.css`，由文档布局引入，避免继续膨胀全局 CSS。

## 搜索与交互

现有客户端和服务端搜索都硬编码为 `english`，与中文内容不匹配。实施时使用 Orama 当前版本支持的中文方案；若中文 tokenizer 不可用，则保留静态搜索并采用可验证的字符级/原文匹配方案，不引入运行时后端。

允许的交互：

- 搜索快捷键和对话框。
- 侧栏折叠、移动抽屉和当前项反馈。
- 任务入口、路径节点和列表行的 hover/focus 状态。
- 右侧目录跟随与轻量阅读进度。
- `prefers-reduced-motion` 下停用非必要运动。

禁止在阅读页启用平滑滚动劫持、鼠标追踪或持续 Canvas 动画。

## 响应式规则

- `>= 1280px`：完整三栏。
- `768–1279px`：左侧栏保留或可收起，右侧栏隐藏为页内目录按钮。
- `< 768px`：单顶栏、模态侧栏、单列内容；任务入口和内容列表不得造成横向滚动。
- 在 375、390、768、1280 和 1440 像素宽度检查布局。

## 性能与静态导出

- 默认使用服务端组件渲染门户和文章列表。
- 只把搜索触发器、移动侧栏、阅读进度等必要模块设为客户端组件。
- ECharts 如被使用，必须仅在门户动态导入，并提供不依赖图表的语义内容。
- 不新增数据库、鉴权、服务端 API 或第三方运行时服务。
- 保持 `output: export` 和 GitHub Pages 路径兼容。

## 错误与空状态

- 没有精选文章时回退到最新发布文章。
- 没有更新时间时隐藏日期，不显示假日期。
- 空分类页展示清晰的规划说明和返回入口。
- 搜索无结果时提供更换关键词和浏览分类的建议。
- 所有链接使用站内相对路径或 `/docs/` 路径。

## 测试与验收

行为采用测试先行：

- 数据规范化、排序、精选回退和状态过滤必须有单元测试并先观察失败。
- 静态输出断言门户包含真实文章标题、中文标签且不包含虚构统计。
- 运行 `npm run lint`、`npm run types:check`、`npm run build`。
- 在本地静态输出中验证导航、搜索、主题切换、侧栏和主要链接。
- 对比 1440×1024 的方案 2/3 参考图和实现截图，修复 P0/P1/P2 视觉差异。
- 检查浅色、深色、375px 手机和 1280px 桌面，不允许横向溢出或控制台错误。

## 非目标

- 用户账号、云端书签和跨设备学习进度。
- 用虚构文章填满门户。
- 将营销首页的所有动画库复制到文档区。
- 在本轮重写全部现有分类内容或部署到 `main`。
