这篇教程适合资料已经散落在聊天记录、收藏夹和文档中的读者。你只需要一个支持 Markdown 或普通文本的文件夹，不需要先购买复杂软件。

完成后，你会得到一个可以被人和 AI 同时理解的最小知识库，以及每周维护流程。

## 第一步：从用途确定目录

不要按“PDF、网页、图片”分类，而要按未来如何使用分类：

```text
knowledge-base/
├─ inbox/          # 尚未处理
├─ projects/       # 正在推进的工作
├─ methods/        # 可重复的方法
├─ references/     # 稳定参考资料
└─ archive/        # 已结束或低频内容
```

目录控制在五到七个。分类过细会让保存资料变成负担。

## 第二步：规定最小元数据

每条正式内容至少记录：

```yaml
---
title: 清楚、可搜索的标题
source: 原始链接或文件
captured: 2026-07-15
tags:
  - 主题
status: active
---
```

元数据只保留以后会用于筛选、核验或维护的字段。

## 第三步：把原文与自己的判断分开

推荐正文结构：

```markdown
## 来源摘要
材料直接表达了什么。

## 我的判断
它为什么重要、与已有知识有什么关系。

## 可执行动作
下一步要测试、复用或确认什么。
```

这样 AI 在后续总结时不容易把你的推测误当成来源事实。

## 第四步：让 AI 只做受约束的整理

```text
请把下面材料整理为知识卡片。
保留原始来源；区分“来源摘要”和“我的判断”；
没有证据的内容放入“待确认”，不要补写事实。
输出 Markdown，不创建新的分类。
```

整理后必须抽查来源链接和关键数字。

## 第五步：建立检索入口

先实现三种最常用的查找方式：

1. 按标题全文搜索。
2. 按标签筛选。
3. 从项目页链接到相关方法和参考资料。

不要一开始就追求复杂向量检索。只有当关键词搜索明显无法满足需求时，再评估嵌入和检索增强。

## 第六步：每周维护 20 分钟

1. 清空或缩小 `inbox/`。
2. 合并重复条目，保留一个主文件。
3. 检查失效链接和过时产品说明。
4. 把已经验证的方法移动到 `methods/`。
5. 为正在推进的项目补一条下一步动作。

## 验证结果

任选一个正在做的项目，应该能在三次搜索或三次点击内找到：背景资料、当前判断和下一步动作。如果做不到，优先调整目录与链接，而不是继续添加工具。
