这篇教程适合已经有 Git 仓库、能运行项目验证命令的开发者。你需要准备一个范围明确、可以通过测试或构建判断结果的任务。

完成后，你会得到一组可审阅的代码变更、验证证据和清晰交接，而不是只得到“代码已经修改”的口头结论。

> 核验日期：2026-07-15。OpenAI 官方将 Codex 描述为可以在本地工具或云端协助读取仓库、修改文件、运行命令和测试的编程智能体。客户端、权限和计划可能变化，请查看 [Codex 官方使用说明](https://help.openai.com/en/articles/11369540-using-codex-with-your-chatgpt-plan/)和 [Codex 用例](https://developers.openai.com/codex/use-cases)。

## 第一步：把需求改写成验收条件

不建议只输入“优化这个页面”。更可靠的任务说明包括：

```text
目标：让移动端导航打开后可以通过 Escape 关闭。
范围：只修改导航组件和对应测试。
验收：
1. Escape 能关闭抽屉；
2. 焦点回到打开按钮；
3. 原有点击关闭行为不回归；
4. lint、类型检查和相关测试通过。
```

## 第二步：让 Codex 先检查项目规则

开始修改前，要求它读取仓库内的 `AGENTS.md`、贡献指南、测试脚本和相邻实现。先确认：

1. 允许修改哪些目录。
2. 项目使用什么验证命令。
3. 工作区是否已有未提交改动。
4. 是否需要独立分支或 worktree。

## 第三步：先复现，再修改

如果任务是修复问题，让 Codex 先说明如何复现；如果是新功能，先写一个会失败的测试或静态断言。

```text
先添加最小回归测试并运行，确认它因为缺少 Escape 处理而失败；不要先改实现。
```

这一步能证明测试真的覆盖了目标行为。

## 第四步：限制修改范围

修改过程中持续检查差异：

```bash
git status --short
git diff --stat
git diff
```

如果出现和任务无关的依赖升级、格式化或文件删除，先停下来确认原因。

## 第五步：运行与风险匹配的验证

最小验证通常包括：

1. 新增或相关测试。
2. lint。
3. TypeScript 类型检查。
4. 生产构建。
5. 必要时进行浏览器交互检查。

不要用“类型检查通过”代替“浏览器行为正确”，也不要用截图代替控制台检查。

## 第六步：审阅交接

要求最终交接包含：

- 修改了什么，以及为什么。
- 哪些文件受影响。
- 实际运行了哪些验证及其结果。
- 仍有哪些已知限制。
- 当前分支、提交或预览地址。

## 验证结果

在接受变更前，自己运行一次关键命令并查看 `git diff`。如果你无法把每一项修改对应到验收条件，就还不应该合并。
